Code Comparison

Password Storage

🚨 Vulnerable Code

# Plain text storage - NEVER do this!
conn.execute('INSERT INTO users (username, password) VALUES (?, ?)', 
            (username, password))

🛡️ Secure Code

# bcrypt hashing - ALWAYS do this!
hashed = bcrypt.hashpw(password.encode('utf-8'), bcrypt.gensalt())
conn.execute('INSERT INTO users (username, password_hash) VALUES (?, ?)', 
            (username, hashed.decode('utf-8')))

Login Query

🚨 Vulnerable Code

# String concatenation - SQL Injection vulnerable!
query = f"SELECT * FROM users WHERE username = '{username}' AND password = '{password}'"
cursor = conn.execute(query)

🛡️ Secure Code

# Parameterized query - SQL Injection safe!
query = "SELECT * FROM users WHERE username = ? AND password_hash = ?"
cursor = conn.execute(query, (username, hashed_password))

Error Messages

🚨 Vulnerable Code

# Reveals username existence - enables enumeration
if not user:
    return "Username not found"
if user['password'] != password:
    return "Incorrect password"

🛡️ Secure Code

# Generic message - prevents enumeration
if not user or not bcrypt.checkpw(...):
    return "Invalid credentials"

Security Recommendations

1. Always Hash Passwords

Use bcrypt, Argon2, or scrypt with a unique salt for each password. Never store passwords in plain text or with fast hashes like MD5 or SHA-1.

2. Enforce Password Complexity

Require minimum length (8+ characters), mixed case, numbers, and special characters. Consider using password strength meters.

3. Implement Rate Limiting

Limit login attempts per IP address (e.g., 5-10 requests per minute) to prevent automated attacks.

4. Use Account Lockout

Lock accounts after multiple failed attempts (e.g., 5 attempts) for a reasonable time period (e.g., 5-30 minutes).

5. Use Generic Error Messages

Return generic error messages like "Invalid credentials" to prevent username enumeration attacks.

6. Implement Multi-Factor Authentication (MFA)

Add an extra layer of security with MFA, especially for sensitive accounts and administrative access.

7. Monitor for Suspicious Activity

Log and monitor login attempts for patterns that may indicate brute force attacks or credential stuffing.

8. Use Parameterized Queries

Always use parameterized queries or prepared statements to prevent SQL injection attacks.
